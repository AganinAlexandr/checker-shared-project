# DocSpectrum v0.1 Action Plan

Статус: draft
Дата: 2026-06-15
Основание: `RP-20260615-001` Opus review

## Цель v0.1

Сделать v0-измерительный цикл воспроизводимее, честнее по метрикам и готовым к расширению корпуса до N>2 объектов.

## Must-fix

### 1. Artifact split

Проблема:

полные generated JSON/паспорта в repo создают шумные диффы и противоречат инфра-слою `E:/output/DocSpectrum`.

Решение v0.1:

- перенести полные generated artifacts в `E:/output/DocSpectrum`;
- в repo оставить скрипты, маленький golden fixture и сводные CSV/README;
- review packet должен ссылаться на output-путь и commit скриптов.

### 2. N>2 pairing

Проблема:

текущая логика pair generation берёт первые 2 объекта по разделу и молча потеряет 3-й объект.

Решение v0.1:

- сделать явную стратегию pair generation;
- поддержать all-vs-all по `section_code`;
- поддержать режим `library_vs_query`;
- явно сохранять `pairing_strategy`.

### 3. Feature statuses

Проблема:

informative subspace и coverage пока не имеют статусов признаков.

Минимум v0.1:

- `measured_zero`
- `missing`
- `not_applicable`

Позже:

- `low_confidence` через `broken_encoding_count`, parse warnings и качество extraction.

### 4. Raw cosine policy

Проблема:

raw cosine по положительным количественным признакам завышает сходство и создаёт floor в aggregate.

Решение v0.1:

- убрать raw cosine из weighted aggregate или резко снизить вес;
- оставить raw cosine диагностическим;
- явно хранить `metric_role = diagnostic|scoring`;
- для scoring использовать jaccard axes и/или centered subvector metrics только после уточнения population.

### 5. Normalization population

Проблема:

centered cosine сейчас считается по смешанному корпусу разных типов разделов.

Решение v0.1:

- явно пометить текущий centered cosine как `global_mixed_corpus_diagnostic`;
- после роста корпуса считать статистики по `section_code`;
- если внутри типа недостаточно объектов, возвращать статус `low_population`.

### 6. Near-match signatures

Проблема:

точные SHA1-сигнатуры страниц/таблиц не ловят близкие, но не идентичные структуры.

Решение v0.1:

- первым добавить bucketing/tolerance для table layout;
- вторым добавить bucketed page signature;
- MinHash/SimHash отложить до большего корпуса и текстового шинглинга.

## Needs human / eval

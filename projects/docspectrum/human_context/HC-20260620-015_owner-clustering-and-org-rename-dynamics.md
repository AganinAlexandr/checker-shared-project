## [opus digest] — это ГЛАВНЫЙ UC (электронный почерк → владельцы)

- **UC: кластеризация орг → ВЛАДЕЛЬЦЫ/группы** по триаде связок: (1) **почерк оформления** (структура+текст — наш валидированный H1/H2-сигнал), (2) **общий ГИП**, (3) **временной паттерн**.
- **Два детектируемых явления (подтверждены временным графиком 20-06, дата_вх поквартально):**
  - **RENAME/дрифт** = временная АНТИ-корреляция (↓↔↑) + общий ГИП + continuity почерка. Кандидаты: **Комтех→АО ССУ№3** (ГИП питанов), **СК Вектор→СтройМонтаж СП** (ГИП рузаев).
  - **Мульти-орг одновременно (один владелец/субподряд)** = КОНКУРЕНТНОЕ существование + общий ГИП/почерк. Пример: Сергеев (Ватага/Гамма/Стройинвест, все 2025).
- **Петля:** валидированный почерк (RP-030..038) ИСПОЛЬЗУЕТСЯ как связка для разрешения идентичности орг → продукт, а не только валидация.
- **Следствие для базы сравнений:** мержить rename-пары (вернуть same-org baseline) + использовать мобильность ГИП → растит H2 из ИМЕЮЩИХСЯ данных сильнее, чем сырое +70-80% объектов.
- **Уровень кластеризации:** из-за внутри-орг конкурирующих групп — кластеризовать на под-орг уровне (group), не предполагать 1 орг = 1 почерк.

## ИУЛ как 4-й сигнал идентичности (human 20-06)

- **ИУЛ (информационно-удостоверяющий лист)** = список проектировщиков организации-ИСПОЛНИТЕЛЯ + подпись ГИП из реестра **НОПРИЗ**. На диске **4908 ИУЛ** (по одному на раздел; извлечены, исключены из секций — корректно, это не ПД-контент).
- **→ 4-й link-type для owner-кластеризации (к почерку+ГИП+temporal): ПЕРСОНАЛ-OVERLAP.** Орги, делящие людей в ИУЛ-ростерах → связь идентичности/владельца/группы.
- **Кросс-реф с НОПРИЗ** (есть как класс): может ловить подлог (проектировщик не в НОПРИЗ / неправдоподобно у несвязанных орг).
- **Caveat (human):** некоторые ростеры ПОДЛОЖНЫ → персонал-overlap = подтверждающий сигнал, НЕ доказательство; сверять с почерком (структура/текст).
- **Билд:** harvest ИУЛ → (орг, ростер[имена], подписант-ГИП) → person×org bipartite → орг-связи; наложить на почерк+temporal+ГИП → кластеры владельцев/групп. Файлы готовы на диске.

## ⚠️ ИУЛ ОТМЕНЯЮТ → ИУЛ = ВАЛИДАЦИОННЫЙ слой, НЕ продакшн-сигнал (human 20-06)

- ИУЛ (иногда ошибочно «УИЛ») есть ВЕЗДЕ (4908), но **регуляторно отменяется** → в будущей ПД его не будет.
- **Следствие:** ИУЛ-персонал-ростеры использовать ТОЛЬКО как **ground-truth для проверки тестовых связей** (подтвердить owner/rename/сеть-кластеры), НЕ как продакшн-фичу.
- **ПРОДАКШН owner-кластеризация = только ДОЛГОВЕЧНЫЕ сигналы:** почерк (структура/текст, H1/H2) + ГИП + temporal (дата_вх ↓↔↑).
- **Главный выигрыш:** если handwriting-кластеры ≈ ИУЛ-персонал-кластеры (на наших тестовых данных) → ДОКАЗАНО, что долговечный метод воспроизводит истину и БУДЕТ работать без ИУЛ. То есть ИУЛ — это калибровочно-валидационный набор (как owner-подтверждённый Комтех→ССУ, Тиволион-4, Ватага-сеть), а не вход модели.

## Регуляторный переход ИУЛ→УКЭП (human 20-06, нормативка)

- **ИУЛ-эра (наш корпус ≤2025):** разработчики разделов подписывали на БУМАГЕ → ростер в ИУЛ → весь пакет подписывал ОДИН ответственный (обычно ГИП) с УКЭП. Поэтому ГИП-на-титуле = цифровой подписант, ИУЛ = реальный (но фальсифицируемый) ростер разработчиков.
- **Отмена ИУЛ — Письмо Минстроя 30-01-2026 №4420-КМ/14.** Причины: недостаточная персональная ответственность; проблемы проверки подлинности подписей; **оформление с подписями лиц, фактически НЕ участвовавших в разработке** (= подлог).
- **УКЭП-эра (с 01-03-2026):** КАЖДЫЙ специалист-разработчик/согласующий подписывает УКЭП-сертификатом; все виды ПД в Главгосэкспертизу; **делегирование МЧД ЗАПРЕЩЕНО**; рекомендована отсоединённая подпись (КриптоАРМ).
- **Следствия для DocSpectrum:**
  1. **Персонал-сигнал МИГРИРУЕТ ИУЛ→УКЭП и УЛУЧШАЕТСЯ:** будущие ПД дают аутентифицированные ПЕР-СПЕЦИАЛИСТНЫЕ УКЭП-подписи (машиночитаемо, по разделу, без делегирования) → durable продакшн-сигнал идентичности/авторства (сильнее фальсифицируемого ИУЛ). Т.е. персонал-связка останется и в продакшне (через УКЭП-метаданные), не только как ИУЛ-валидация.
  2. **Регуляторная релевантность DocSpectrum:** Минстрой прямо называет проблему «подписи лиц, не участвовавших в разработке» — это ровно то, что детектит почерк (реальный автор vs номинальная подпись). Почерк = контентная проверка авторства, ДОЛГОВЕЧНА через обе эры.

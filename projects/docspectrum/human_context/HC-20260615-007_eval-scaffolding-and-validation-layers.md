## [human]

- Весь надстроечный слой (градиентные объекты, две оси, пары одного дома, ТЭП-корреляции) — это **эшафот над тестовыми разделами, нужный только для правильной отладки алгоритмов**, не сам продукт. Если отладить алгоритм на этих удобных тестовых разделах, он с большей вероятностью будет правильно работать на любых PDF.
- По заметному количеству этих объектов (скорее всего по большинству) позже будет **ПД после экспертизы** — дополнительный слой сравнения. Состоит из исправлений: (1) **инструментальных ошибок** проектировщиков (систематические, от инструментов/шаблонов); (2) **хаотичных зевков** любых исполнителей (случайные, превращающиеся в ошибки).
- Для капремонта кроме «обучающей выборки» можно организовать и **«тестовую выборку»** любого нужного состава — тот же проектировщик и тот же вид работ, либо другие проектировщики в разной степени различия.
- Полного («совсем любого») разнообразия по разделам предоставить нельзя, но **имеющиеся варианты — это уже многое**.

## [opus digest]

Принципы для фронта eval/calibration (RP-011 и далее):

- **Eval = ВАЛИДАЦИЯ механизма, не ПОДГОНКА.** Опасность — переобучить веса/пороги под текущий узкий корпус (18 разделов, один орг, до экспертизы). Калибровка должна давать **общие, обоснованные** параметры; критерий — ранговый/sanity, не максимизация фита под известный градиент. Эшафот **валидирует** доменно-нейтральное ядро, но НЕ входит в него (`CORE_DOMAIN_LEAK`).
- **Что валидируем — МЕХАНИЗМ** (дисконтирует ли DF/IDF boilerplate; ловит ли jaccard ось содержания; разделяет ли метрика две оси A/B — HC-005). Отладка на корпусе с известной истиной → высокая уверенность в исправности механизма.
- **Обобщение на «любой PDF» — обоснованная ЭКСТРАПОЛЯЦИЯ от проверенного механизма, не доказанная универсальность.** Полного разнообразия разделов нет; имеющейся вариативности достаточно для валидации механизма, но широкую кросс-доменную/кросс-орг универсальность доказывает только более разнообразный набор.

Доступные слои валидации:

- **обучающая выборка** — текущий корпус (один орг, капремонт, до экспертизы);
- **тестовая выборка** — собираемая с управляемой дистанцией проектировщиков (тот же / другие проектировщики разной степени отличия) → проверка обобщения + полигон оси «электронного почерка»; **калибровать на train, валидировать на held-out test**;
- **до/после экспертизы** — ground-truth для оси `version_diff` (comparison_result_model §18): что изменилось = что исправили. Структура дельты: **инструментальные** ошибки (систематические, повторяются между работами проектировщика → детектируемы по DF, это и есть почерк = СИГНАЛ) + **хаотичные** зевки (случайные, одноразовые, DF=1 → шумовой пол).
  - **Объективный бинарный чекпойнт (вторичный):** после экспертизы ТЭП одинаковых домов должны стать ИДЕНТИЧНЫМИ (если до экспертизы различались) → `address_tei_consistency` переходит `inconsistent → consistent` (напр. 1680/1684). ТЭП одного дома однозначны → чистый не-шумный сигнал, что версионная ось ловит реальное исправление. Вторичен: ТЭП = eval-контекст/3-й порядок (HC-006), не ядро → подтверждающий QC-сигнал, не главный тест механизма.

Реалистичные ожидания (ревью): DocSpectrum хорошо подходит для **систематических** (инструментальных) расхождений — они и есть почерк; **хаотичные** ловить почти бесполезно (случайны) → на before/after ждать структурированный сигнал + шум, не чистый сигнал; recall по хаотичным закладывать низким.

## [opus digest]

- **dev-ГИП НЕ монолитен — есть субподрядная цепочка.** 4-титульный документ кодирует ДВА ГИП:
  - **ГИП субподрядчика = РЕАЛЬНЫЙ автор** (кто фактически разрабатывал) → носитель почерка в контенте;
  - **ГИП проектировщика-заказчика = контрактный** (головной/подписант) → может быть «подпись-в-аренду» (вариант signature-for-hire из HC-009).
- **Цель атрибуции почерка = ГИП СУБПОДРЯДЧИКА** (реальный автор), не заказчик. `доп_2` = единственный ГИП-проектировщик в таблице; для 4-титульных (субподрядных) реальный автор (субподрядчик) в таблице ОТСУТСТВУЕТ → только **экстракция с титулов** (HC-011). Для 2-титульных доп_2 = и есть проектировщик-автор.
- **Кол-во титульных (2 vs 4) = ДЕТЕКТИРУЕМЫЙ сигнал субподряда** (детектор даёт `title_anchor_pages`/`cover_end`). В текущих выгрузках почти все 2-титульные (RSPK/NK = прямые проектировщики; 4 титула лишь 1/451). Субподряд проявится в более широком кандидатском корпусе (кровельные объекты разных орг).
- **Следствие для модели почерка (уточняет HC-009/HC-012):** при 4-титульной структуре орг/ГИП-почерк надо атрибутировать СУБПОДРЯДЧИКУ; заказчик-ГИП — отдельный слой (контрактный, возможно без отпечатка). Provenance-first (HC-012) + это = полная цепочка авторства: third-party (производитель/софт/вендор) → субподрядчик (реальный автор) → заказчик (контракт/подпись).
- **Следствие для корпуса GIP:** `доп_2` (и проектировщик/генподрядчик) = стартовый пул; таблица **может содержать ошибки** (не всегда актуально вносилась) → **ТИТУЛЫ авторитетны для И орг, И ГИП**. Финальная атрибуция — с титулов после выгрузки (+ распознать 2 vs 4 титула → пометить субподрядные → взять субподрядчик-ГИП как истинного автора). xlsx-метки в `gip_corpus_candidates_v0.csv` = ТОЛЬКО для отбора пула, не ground-truth.

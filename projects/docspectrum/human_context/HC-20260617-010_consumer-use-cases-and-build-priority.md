Продуктовые задачи потребителя, которые библиотека типовых элементов должна уметь реализовать. Приоритет = порядок следования. Связано с T-20260617-016 (дизайн библиотеки), HC-20260615-008/HC-20260617-009 (ось C/почерк).

## [human] — 4 задачи потребителя (по приоритету)

1. **Шеф проверяет проект сотрудника** (есть библиотека разделов своей орг. с извлечёнными типовыми элементами):
   - 1а. соответствует ли раздел почерку ОРГАНИЗАЦИИ;
   - 1б. соответствует ли почерку самого ИСПОЛНИТЕЛЯ (буквально — «не списал ли сын домашнее задание»);
   - 1в. какую часть материала исполнитель создал сам.
2. **Друзья передали раздел** («возьми на вооружение качественный материал») → предварительное сравнение со своей базой.
3. **Проверка прав/собственности:** исторически права на проект прописывались в договорах; собственник проектных решений хочет проверить чужую ПД на использование ЕГО материалов.
4. **Разработка нового раздела:** сравнить имеющиеся шаблоны и выбрать наиболее подходящий.

**1б — ПАРКУЕТСЯ:** данных по следу исполнителя сейчас НЕТ; **ГИП ≠ исполнитель уровня раздела**. Human позже достанет разделы с гарантированными исполнителями. До тех пор за 1б не браться.

## [opus digest] — карта задача→возможность и последовательность сборки

Маппинг:
- 1а → покрытие раздела ОРГ-ОТЛИЧИТЕЛЬНЫМИ типовыми элементами (P2 орг-IDF/отличит. слой);
- 1в → coverage → typical_share vs original_residual (P1);
- 1б → профиль per-исполнитель + сверка ОСТАТКА (P3 co-occurrence + P4 метка + P1 остаток) — ГЕЙТ на данные;
- 2 → section-to-library similarity + coverage (P1 + существующий движок);
- 3 → заимствование = ФОРМА + near-same СОДЕРЖИМОЕ моих отличительных материалов (near-match + content + P2);
- 4 → ранжирование каталога форм по близости (P1 + similarity, ТЗ §7.2).

Ключевое различение, которое выявили задачи: **форма ≠ содержимое ≠ остаток.** 1а/4 = совпадение ФОРМЫ (норма, не криминал); 1б = совпадение СОДЕРЖИМОГО ОСТАТКА с чужим (вот это «списал»); 3 = ФОРМА+СОДЕРЖИМОЕ. Остаток (не объяснённое шаблонами) — где живут и оригинальность, и списывание. Использование орг-шаблонов ОЖИДАЕМО; «списал» = совпал ОСТАТОК, не шаблон.

**Последовательность сборки (= приоритет human):**
- **Сборка 1 — библиотека v0 = каталог + coverage(P1) + орг-отличительность(P2).** Закрывает 1а, 1в (приоритет #1) + попутно 2 и 4. **P1/P2 — НЕ задел, а необходимое для #1 → строить ПРЯМО в v0** (v0 = каталог + покрытие + орг-IDF сразу, не каталог-сначала).
- **Сборка 2 — near-match.** Закрывает 3 (приоритет #3) + достраивает детекцию заимствования оси C; near-match на критическом пути, окупается дважды.
- **Сборка 3 (гейт на данные) — 1б**, когда появятся разделы с метками исполнителей.

Границы: корреляция ≠ юр-доказательство (ТЗ §5.2, «информируем не караем») — особенно для задач 1б и 3.

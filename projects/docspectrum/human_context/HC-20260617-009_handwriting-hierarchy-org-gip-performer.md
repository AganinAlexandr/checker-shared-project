## [human]

- Два РАЗНЫХ ГИПа: **ГИП экспертизы** (Аганин — это сам human, надзор/проверка, в документах почерка НЕ оставляет → для оси C нерелевантен) vs **ГИП разработчика** (`ГИП_по`, стоит на ТИТУЛАХ разделов, ведёт работу проектной организации → носитель почерка).
- В реестре (devel-ГИП): РСПК → **Оборина**, НК-Инжиниринг → **Траскунова** (разные ГИПы и орги).
- dev-ГИП — член НОПРИЗ, имеет право подписи ПД. Отсюда при имени ГИПа на документах разных оргов — три варианта:
  1. **ГИП реально вёл/шаблонировал** → хорошая корреляция к его почерку;
  2. **ГИПу заплатили за подпись** («свадебный генерал») → корреляции нет (реальные авторы разные);
  3. **смешанный** — подписал И поделился шаблонами → только форменное/шаблонное совпадение.
- Позитивные ГИП-кейсы (один dev-ГИП на разные орги: Ширяев/Сергеев/Ефимов/Локтев/…) — это ДРУГИЕ объекты, НЕ УУиР → отдельная отработка: берём их объекты → новые выгрузки → та же схема.
- Внутри орга: ГИП знает шаблоны/исполнителей/субподрядчиков, раздаёт работу по загрузке (своя команда + субподрядчики). Капремонт удобен — «может почти каждый по шаблону». Под одной шапкой один раздел делают разные люди, но при постоянной работе и общих шаблонах. Для простых объектов ГИП сам раздаёт шаблон (оформление и объём тем известны).
- **Вывод:** для капремонта внутри ОДНОГО орга корреляция должна быть заметна ДАЖE для разных исполнителей. По многим образцам орга можно померить разброс личного следа исполнителя и даже разделить следы — если это **исполнители** (оставляют след), а не **заполнители** (механически заполняют шаблон).

## [opus digest] — иерархия почерка (4 уровня) и привязка к метрике

- **L0 — экспертный ГИП (Аганин):** нерелевантен (нет отпечатка в документах).
- **L1 — ОРГАНИЗАЦИЯ:** общие шаблоны → высокий within-org «пол» сходства, разделяет орги (РСПК↔НК gap 0.27-0.67, RP-015). Капремонт шаблонно-простой → within-org корреляция держится ДАЖE для разных исполнителей.
- **L2 — dev-ГИП (титул):** раздаёт шаблоны/назначает; cross-org same-ГИП → **3 варианта ↔ слои метрики**: вариант 1 (автор) = ВЫСОКОЕ whole-section idf_similarity; вариант 2 (подпись-за-деньги) = НИЗКОЕ (как cross-org baseline); вариант 3 (смешанный) = только ШАБЛОННОЕ/форменное совпадение = слой **типовых элементов** (форма совпала, содержимое разное — как мост ИОС5.4.1 в RP-015). → **библиотека типовых элементов = дискриминатор варианта 3**. Позитивный тест требует выгрузки объектов этих ГИПов (не-УУиР).
- **L3 — ИНДИВИД. исполнитель:** личный след = **IDF-остаток НАД орг-шаблонным полом** (IDF давит высоко-DF шаблон, поднимает редкие сущности = личная вариация). Разделение исполнителей = остаточная кластеризация по-секционно (**NK-кластеры Codex = первый срез**); **cluster co-occurrence по секциям** разводит объект-серию/пакет (кластер в многих секциях) vs секц.-специфичного исполнителя (кластер в одной секции). **исполнитель** = есть стабильный остаточный кластер (отделимый след); **заполнитель** = чистый шаблон, остатка нет.

## Границы (КРИТИЧНО)

- Всё это — **корреляционное свидетельство, НЕ юридическое доказательство** авторства/подписи (ТЗ §5.2); философия «информируем, не караем».
- dev-ГИП **извлекаем из титульных страниц** (в экспорте) → в-документная верификация ground-truth, не только внешний xlsx; кандидат на мелкий tool.
- **near-match — предусловие** L2/L3: вариант 3 и личный след = near-identical, точными сигнатурами не ловятся.

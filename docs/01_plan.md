# План разработки

## Шаг 0. Входной контроль PDF

Цель: определить пригодность раздела для детальной проверки.

Основные проверки:
- наличие текстового слоя;
- качество кодировки;
- `page_kind`: vector / hybrid / scanned / text_outlined_suspect;
- наличие OCR-потребности;
- наличие и качество СПДС-рамок;
- таблицы, ячейки, области с векторной графикой;
- возможность копирования и поиска в соответствии с требованиями Приказа 783.

Основной поставщик структурных данных-кандидат: `pdf-structure-explorer`.

## Шаг 1. Подготовка области поиска и поиск первого блока BODY

Цель: отсеять случаи/зоны, где детальный поиск первого блока некорректен или не нужен, и затем найти начало body-корпуса.

### Шаг 1.1. Проверка существенности битой кодировки

Сначала проверяется, имеет ли раздел существенную часть текста с битой кодировкой.

Правило:
- если битая кодировка затрагивает 1-2 пункта, детальная проверка может продолжаться, а по этим пунктам выставляются отдельные замечания;
- если битая кодировка затрагивает больше 1-2 пунктов или существенную часть раздела, выставляется общее замечание по битой кодировке и нарушению требований Приказа 783; дальнейшая детальная проверка может быть остановлена или переведена в ограниченный режим.

### Шаг 1.2. Поиск и отсечение титульной зоны

После проверки кодировки ищутся титульные страницы: обложки и титулы.

Обычно их две, но при наличии субподрядчика может быть четыре или более титульных страниц.
Титульная зона исключается из пространства поиска первого блока.

### Шаг 1.3. Поиск первого блока BODY

После отсечения титульной зоны поиск первого блока ведется только в суженной области документа.

Базовое определение блока:
`block = marker + heading + сразу body_text`.

Если после heading сначала идет вложенный ряд, таблица, подзаголовок или иная промежуточная структура, верхний элемент не считается block без дополнительного правила поглощения вложенного уровня.

## Шаг 2. Построение и выбор корпусов по зонам

Цель: построить кандидаты корпусов TOC и BODY и выбрать правильные.

Принцип:
- TOC-зона расположена до правильного BODY-корпуса;
- TOC обычно состоит из пар `marker + heading`, возможно в таблице;
- BODY состоит из блоков;
- таблицы нигде не являются источником блоков;
- таблицы могут быть источником только пар для TOC-корпусов;
- сравнение с ПП87 используется как смысловой усилитель, но не заменяет структурные признаки.

## Шаг 3. Выставление замечаний

Цель: сформировать item-level verdicts и доказательства.

Базовые статусы:
- `present`;
- `missing`;
- `wrong_text`;
- `wrong_marker`;
- `wrong_version`;
- `not_applicable`;
- `unchecked`.

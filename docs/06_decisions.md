## DEC-20260613-003: Этапный протокол обмена (теги + триггеры человека)

Последствия:

## DEC-20260614-004: Collaboration workflow v2 (ретро 2 дней)

Статус: accepted
Дата: 2026-06-14

Контекст:
Ретро после 2 дней (human — 15 лет PM). Схема рабочая, но нужна формализация: доменные основания решений терялись, когда пояснения human шли только в одном чате; вскрыт пробел — код Opus был ЛОКАЛЬНО (не пушился) → Codex ревьюил описания, не код. Консенсус opus+codex+human.

Решение:
1. **Модель — ГЕНЕРАТОР-РЕВЬЮ** (не «равные_роли» — неустойчивое равновесие, скатывается к одному краю). Checker: **Opus generator / Codex reviewer+contract-owner / human strategic owner**. DocSpectrum: **Codex generator / Opus reviewer / human owner**. Драйвер — по рабочему участку; Codex-генератор для контрактов/словарей/shared-структуры/eval/version-overlay-policy. ПРАВИЛО: в одном проекте не два независимых генератора без явного разделения зон.
2. **Код — на GitHub по ссылкам, НЕ в shared.** opus_checker → `github.com/AganinAlexandr/pp87-checker @ heading-template-matching`; пуш после КАЖДОГО заметного коммита. Shared = координатный слой (ссылка на commit + что проверено + входные данные + метрики + выводы + открытые вопросы), не монорепа кода.
3. **Канал human_context:** `discussion/human_context/HC-YYYYMMDD-NNN_<topic>.md`; формат блоков `[human]` / `[opus]` / `[codex]`; ТОЛЬКО сообщения, меняющие правило/приоритет/термин/метрику/тип замечания/граничное условие. Digest делает агент, ПОЛУЧИВШИЙ сообщение (триггер «занеси в human_context» или сам распознаёт существенность).
4. **Checkpoint = review packet:** коммит, входные данные, главные reason-коды, регрессии, что НЕ трогалось, какое доменное правило использовано/изменено. Атомарно (пуш кода + чекпойнт в одном ходе).
5. **Запись по слоям:** threads (сырое) → 06_decisions (финал) → contracts (данные/reason/метрики) → task_board (план/статус) → human_context (доменные вводные).

Последствия:
Ревью становится настоящим (по коду); доменные основания фиксируются; роли явные; Checker/DocSpectrum — зеркальный эксперимент.

Ссылки:
- `discussion/threads/T-20260613-001`, `T-20260613-002`
- `discussion/human_context/` (HC-записи)
- `registries/repositories_registry.csv`

## DEC-20260615-006: crc32-адресация bundle (doc_<crc32>) как канон связи

Статус: accepted
Дата: 2026-06-15

Контекст:
Папки выгрузок `pdf-structure-explorer` ранее переименованы с random-чисел на `doc_<crc32>` именно ради простой и надёжной связи документ↔bundle. Общий каталог `exports` мутируется параллельно (Checker + DocSpectrum, 440+ bundle, рост): резолв по `file_name` нестабилен — коллизии имён (один file_name у разных crc), а также временная архивация/добавление сдвигают результат (наблюдалось: знаменатель Checker 1269→1215 между прогонами). Codex в параллельном чате независимо столкнулся с той же задачей адресации.

Решение:
Канонический способ связи раздел↔bundle — **crc32 содержимого исходного PDF** → папка `doc_<lower(crc32)>`:
- crc32 считается от ИСХОДНОГО PDF (стабилен, не зависит от мутаций exports);
- ключ `doc_<crc32>` уникален → новые выгрузки с тем же именем НЕ перекрывают;
- поиск bundle РЕКУРСИВНЫЙ (вкл. `exports/архив/...`) → переживает архивацию;
- `file_name`/`document_links` — лишь fallback.

Реализация Checker: `tools_heading/toc_pairs_from_cells.py::bundle_path(crc)` (+ `_bundle_index`, рекурсивный); `item_level_toc.py::resolve_by_crc/local_pdf_by_name`; резолв в `item_level_toc`/`body_item_verdicts` теперь crc-FIRST (commit opus_checker@18a400d). Фиксация тестсета: `build_checker_manifest.py` → объект+раздел+ПОДРАЗДЕЛ→crc32→bundle.

Последствия:
Резолв детерминирован и иммунен к параллельной работе в общем exports (подтверждено: TOC 822/1215 стабильно run-to-run). Остаточный риск — физическое удаление нужного `doc_<crc>` (тогда — снапшот в приватный каталог). Конвенция общая для Checker и DocSpectrum.

Ссылки:
- `opus_checker@18a400d` (pp87-checker, branch heading-template-matching)
- `tools_heading/toc_pairs_from_cells.py`, `tools_heading/build_checker_manifest.py`

## DEC-20260621-007: Collaboration workflow v2.1 (ретро 10 дней, DocSpectrum)

Статус: accepted
Дата: 2026-06-21

Контекст:
Ретро после ~10 дней. Схема DEC-20260614-004 (генератор-ревью) подтверждена и продуктивна
(эксперимент C + совместный assessment HC-20260621-018 — два независимых среза opus+codex сошлись).
Базовые роли НЕ меняем; уточняем границы, чтобы снять копипаст-релей, контекст-расход и тихий дрейф
«тестирование↔исследование». Консенсус human+codex+opus (кросс-ревью 2026-06-21).

Решение (уточняет DEC-20260614-004; роли из него сохраняются):
1. **Shared вместо релея.** Полные RP/HC/probes/обсуждения агенты читают НАПРЯМУЮ из shared. Через
   human передаётся только: (а) доменная истина; (б) стратегическое решение; (в) короткое уведомление
   (что зафиксировано / commit / что требуется от human). Human снимается с роли транспортного канала.
2. **Generator-ready gate.** До реализации в задаче зафиксированы: цель + связь с ТЗ; **ТИП работы
   (валидация / исследование данных / продуктовая сборка)**; входы и expected outputs; acceptance/reference;
   ограничения/not-touched. Задача уходит генератору только при заполненном гейте.
3. **Обязательно в каждом RP два вопроса:** «Что пакет проверяет — алгоритм / данные / продуктовый
   сценарий?» и «Какую задачу ТЗ он валидирует?». Ловит дрейф сразу, а не на бэкенде.
4. **Probes Opus = reviewer-generated, маркируются.** Probe пишется МИНИМАЛЬНО и ТОЛЬКО когда без данных
   нельзя принять решение / нужно установить reference для гейта (probe производит reference, не дубль-прототип).
   Если probe влияет на вывод — Codex ОБЯЗАН независимо проверить его допущения при продуктизации
   (закрывает риск частичного саморевью Opus; прецедент — баг 1284_24 в probe Opus, пойман Codex).
5. **Укрупнить milestones.** Один RP = законченный методический/пользовательский слой; мелкие
   reviewer-fix → follow-up-коммит внутри прежнего RP (без нового RP). **ОГОВОРКА (opus):** каждый
   крупный milestone ОБЯЗАН нести `--assert-reference` + тесты, иначе укрупнение = слепое доверие
   (большой пакет иначе не воспроизводим атомарно).
6. **Дайджест сокращает КОММУНИКАЦИЮ, не верификацию (opus).** После RP — короткий digest, если находки
   уже в shared. Но ревьюер ВСЁ РАВНО независимо прогоняет assert-reference + тесты + артефакты. Как только
   [AGENT-OK] станет штампом без воспроизведения — несущий плюс схемы (независимая проверка) испарится.
7. **Фазовый стратегический checkpoint.** Совместное обсуждение: перед новой крупной аркой; после её
   завершения; либо после 3–5 RP, если работа пошла по инерции.
8. **Стоп-критерий ДО начала арки.** Заранее определять, какой результат достаточен и когда доводка не
   окупается (эксперимент C подтвердил ценность — owner остановил доводку depth по «данных мало»).
9. **Owner-briefing для значимых решений:** что измеряем / почему так / что означает результат / чего НЕ
   доказывает / расшифровка новых обозначений. Позволяет human углубляться выборочно, не перечитывая поток.
10. **Асимметрия расхода естественна — не выравнивать** (генератор держит код/данные/тесты/интеграцию;
    ревьюер — один диапазон). Изолированные вспом-инструменты / разметка / ground-truth / аудит данных
    могут идти Opus как ЯВНО согласованные подзадачи, но владелец-интегратор DocSpectrum — один (Codex).

Сохраняем без изменений: Codex — единственный основной генератор DocSpectrum; Opus — независимый ревьюер
с ОБЯЗАТЕЛЬНЫМ воспроизведением; human — стратегический владелец + источник доменной истины; committed
diff — предмет ревью; финальное обсуждение (предотвращает дрейф, пересматривает приоритеты).

Итоговая цепочка:
`совместное обсуждение → committed HC/spec → generator-ready gate → Codex milestone (+assert-reference+тесты)
→ Opus independent review (реальное воспроизведение; дайджест-коммуникация) → human decision → фазовая ретроспектива`.

Последствия:
Меньше копипаста и повторного контекст-чтения; human — на понимании/направлении/доменной коррекции, не на
транспорте; дрейф тест↔исследование ловится в каждом RP; верифицируемость крупных пакетов защищена
(assert-reference+тесты); базовые роли DEC-004 неизменны.

Ссылки:
- `DEC-20260614-004` (база v2, уточняется)
- `projects/docspectrum/human_context/HC-20260621-018` (совместный assessment + validation-backlog)
- `discussion` DocSpectrum 2026-06-21 (кросс-ревью opus+codex по workflow)

## DEC-20260629-008: Progressive dual-review workflow (профили DOC-2 / CODE-2 / CODE-3)
Статус: accepted
Дата: 2026-06-29

Контекст:
Из нити `T-20260629-001`. Испытаны два режима ревью: A (генератор + основной full-context ревьюер)
и B (генератор + несколько blind focused-ревьюеров). Пробел режима B: никто не проверял достаточность
focused-пакета, выбранного генератором; одинаковый пакет создаёт общий риск пропуска контекста. Цель —
progressive disclosure: независимые blind focused-отзывы + аудит качества пакета основным ревьюером,
с учётом изоляции контекста и стоимости.

Решение (уточняет DEC-20260614-004 и DEC-20260621-007; их роли/воспроизведение сохраняются):
1. **Phase-0 design-gate** обязателен до методических изменений (метрики/benchmark/ground-truth/
   threshold/инъекция/verdict-правила/состав данных). Рефакторы и реализация принятого контракта — без gate.
2. **Изоляция focused-стадии — ФИЗИЧЕСКАЯ:** свежая сессия на КАЖДЫЙ review-cycle; внутри цикла Phase 2→3 —
   ТА ЖЕ сессия (`same_reviewer_context_delta`); focused-первый пасс — в контролируемом по входу чате, НЕ
   агенте; браузинг/доступ к репо на слепом пассе выключен. `cross_instance_context_delta` — только диагностика.
3. **Вопросы A/B асимметричны** при общем frozen snapshot + обязательном unguided sweep первым шагом.
4. **`[AGENT-OK]` расщеплён:** `reasoning-OK` (focused→full аудит) **∧** `reproduction-OK` (воспроизведение
   тестов/артефактов).
5. **Operational profiles — по ТИПУ ДЕЙСТВИЯ, не по проекту:**
   - **DOC-2** (документ/методология/данные/самодостаточный CSV): A progressive-чат + B blind-чат (2 ревьюера);
     полный итог — только если песочница A реально воспроизводит весь заявленный evidence.
   - **CODE-2** (экономичный): A reasoning + T technical verifier — 2 прохода + воспроизведение, БЕЗ 2-го blind;
     только низкие ставки; явно не выдавать за «1+2 blind».
   - **CODE-3** (надёжный): A reasoning + B blind focused + T technical verifier; `[AGENT-OK] =
     reasoning-OK ∧ reproduction-OK`. `reproduction-OK` выдаёт не generator, а T, независимо перезапускающий
     генераторский детерминированный автотест-сьют. T — техническая роль, а не третья blind reasoning-семья;
     отдельная третья модельная семья нужна лишь при сомнении в тестах/high-stakes. Два blind reasoning-review
     выполняются на каданс связного блока.
6. **Третий reasoning-reviewer (R3 / 1+3; не T technical verifier)** — не по умолчанию: high-stakes /
   неразрешимый конфликт A/B / подозрение на общий framing bias.
7. **Правило сходимости:** дискуссия — ИСКЛЮЧЕНИЕ, не обязательная фаза. Дефолт: сошлись → применили → фриз.
   Дискуссия запускается только при неустранимом evidence-конфликте → по природе `[NEEDS-HUMAN]`, короткий
   owner-таймбокс. Уникальная находка с evidence не подавляется отсутствием консенсуса (Phase 4 reconciliation,
   а не дискуссия).
8. **Per-project default (owner choice):** sorting_center → CODE-3 с 02-07 (до этого без серьёзной работы);
   DocSpectrum → CODE-3 (пилот = валидация схемы на реальном review-cycle); pp87-checker → DOC-2 на стадии
   ручной разметки/golden, CODE-3 + 2-я независимая blind-семья при возобновлении ревью кода.

Последствия:
Закрывает пробел режима B (достаточность пакета измеряется дельтой A focused→full). Сохраняет независимость
двух семей (CODE-3/DOC-2) при контролируемой стоимости. Не отменяет базовые роли DEC-20260614-004 /
DEC-20260621-007. Остаточные практические вопросы (реализация capability isolation на конкретных платформах;
reviewer-calibration suite на реальных исторических дефектах) — в follow-up-нить.

Принято по результату DocSpectrum CODE-3 пилота `CODE3_DS_GOLDEN_20260629_001`:

- A focused→full обнаружил реальные дефекты контракта, а correction-pass подтвердил их закрытие;
- B независимо добавил риски stale-artifact/version binding и переносимости;
- T-попытка без запуска отклонена как `NOT-EXECUTED`; `reproduction-OK` принят только после физического
  запуска manifest/tests/anchors/mutation probes в отдельной Python 3.12/Linux среде;
- итоговый гейт: `REASONING-OK-CORRECTION ∧ REPRODUCTION-OK`.

Два обязательных operational-уточнения пилота:

1. full-context порция должна содержать явную карту claim/anchor→RP/source; неполная карта породила ложные
   N1/N2, позднее отозванные A;
2. T обязан вернуть машинные признаки исполнения (runtime, return codes, hashes, generated report). Анализ
   кода, условное «должно пройти» или непересчитанный manifest не дают `reproduction-OK`.

### Cost-control profile после пилота

1. **CODE-3 применяется только к крупным методическим/архитектурным чекпойнтам.** Рутинная реализация
   принятого контракта по умолчанию идёт через CODE-2; профиль выбирается по риску действия.
2. **Карта `claim → artifact → RP/source` — обязательный generator-ready артефакт пакета.** A не должен
   восстанавливать трассировку раскопкой всей истории; A/B/T используют одну зафиксированную карту.
3. **Full-context для A = дельта к focused-пакету + адресно запрошенные материалы.** Повторная передача кода,
   evidence и истории «на всякий случай» запрещена, если они уже доступны в той же progressive-сессии.
4. **Компактность A относится к выходу, не к глубине проверки.** Обязательный ответ: findings, verdict,
   claim boundary и при необходимости requested context; внутреннее reasoning не заменяется кратким просмотром.
5. **T по умолчанию выполняется независимым CI на чистом checkout.** Предпочтителен machine-generated report
   с manifest hashes, runtime, commands/return codes, tests/artifacts и output hashes. Скрипт генератора без
   независимого исполнения остаётся generator self-test и не закрывает `reproduction-OK`.
6. **Correction-pass минимален:** только changed diff, таблица закрытия находок и необходимая traceability-дельта;
   полный контекст повторно не передаётся.

Экономия достигается сокращением дублируемой передачи и формата вывода, но не ослаблением независимости A/B/T.

| Роль | Первичный | Альтернативы | Правило |
|---|---|---|---|

#### Калибровочный вывод по Gemini

- **B допустим:** независимо обнаружены нестрогий `equal` и stale-artifact риск.
- **B требует reconciliation:** корректный fail-closed механизм ошибочно повышен до критического silent masking;
  замечание о list-filter дубликатах не учитывает, что `len(matches) != 1` уже fail-closed.
- **A запрещён:** в A-проходе потеряна собственная типовая находка, необоснованно одобрены snapshot bands,
  переутверждена изоляция holdout и пропущен центральный relational GIP gap.
- Номинальный размер окна не используется как гарантия внимания. Для packaged B/R3 действует наш
  **операционный лимит 32–64 KB полезного контента**; это packaging policy, а не утверждение о физическом
  контекстном окне модели.
- Любые заявления модели о сохранении выводов в «системной подсказке» между независимыми чатами считаются
  недействительными: роль и контекст подаются заново в каждой сессии.

#### Упаковка для packaged B/R3

1. `claim → artifact → RP/source` помещается в начало.
2. Затем идут diff, ключевые файлы и асимметричные вопросы.
3. Остальной контекст передаётся только адресно; нельзя рассчитывать, что конец большого пакета будет удержан.
4. Размер пакета не определяет семейство B: семейства ротируются для независимости, а пакет всегда укладывается
   в операционный потолок.
